## 前端对后端接口需求规范

### 1. 文档位置与用途

- **存放位置**：在项目根目录下的 `docs` 文件夹中，专门用于存放前端向后端提出的接口需求文档。
- **核心用途**：前端开发人员在此文档中明确描述需要后端新增或修改的接口需求，包括接口功能、请求参数、期望的响应数据结构等。

### 2. 格式规范

- **必须使用 Markdown 格式**：所有需求文档必须以 `.md` 为扩展名，确保内容结构清晰、易于后端开发人员阅读。
- **推荐结构**：
  - **需求背景**：说明为什么需要这个接口（业务场景）
  - **接口定义**：
    - 接口名称与用途
    - 请求方法（GET/POST/PUT/DELETE）
    - 请求路径
    - 请求参数（Headers、Query、Body，包含字段类型和必填项）
  - **期望响应**：
    - 成功响应的数据结构（包含字段类型、示例值）
    - 错误响应的可能情况及错误码
  - **业务逻辑说明**：特殊处理需求、边界条件等

### 3. 协作流程

- **前端提出需求**：在 `docs` 文件夹中创建或更新需求文档，详细描述需要的接口功能及数据结构。
- **后端实现接口**：后端开发人员根据文档中的需求描述，开发新接口或修改现有接口的返回值结构、字段命名及业务逻辑。
- **联调与反馈**：接口开发完成后，前后端进行联调测试，如有问题及时反馈并更新文档。
- **迭代更新**：如需求变更或发现新问题，应及时更新对应文档，保持前后端信息同步。

### 4. 最佳实践

- **文档命名应具有语义性**（如 `submit-lottery-topic.md`），便于快速定位和理解需求内容。
- **提供具体的 JSON 示例**：包括请求示例和响应示例，减少理解偏差。
- **一个需求一个文档**：如涉及多个独立接口需求，建议为每个需求创建独立文档；如属于同一功能模块的多个接口，可在同一文档中分章节说明。
- **标注优先级**：如需求紧急，可在文档开头标注优先级（高/中/低）和期望完成时间。
